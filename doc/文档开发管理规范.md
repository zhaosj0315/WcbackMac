# 文档开发管理规范

## 1. 适用范围

本规范适用于项目根目录 `readme.md`、`doc/*.md`、示例输出说明、审计报告、清理报告和用户使用材料。代码、数据库、导出结果和第三方参考源码不属于本文档规范的直接编辑对象。

## 2. 基本原则

1. 代码为准：文档必须适配当前代码，不能用计划、推测或历史实验结论替代当前实现。
2. 可验证：文档中的命令必须能在当前目录找到对应脚本或入口。
3. 不夸大：未完整验证的功能必须标注“实验性”“部分支持”或“当前不推荐”。
4. 保留边界：涉及微信、macOS 权限、LLDB、缓存图片、语音视频等功能时，必须说明前置条件和失败边界。
5. 单一事实源：同一主题只保留一份主文档，历史计划、阶段报告、旧审计结论不长期保留。

## 3. 文档分类

| 类型 | 文件 | 维护要求 |
|------|------|----------|
| 项目入口 | `readme.md` | 只写当前稳定入口、核心命令、风险边界 |
| Mac 使用 | `doc/Mac版本使用指南.md` | 面向使用者，给可执行流程 |
| Mac 状态 | `doc/Mac版本功能总结.md` | 面向维护者，列功能状态和限制 |
| 技术记录 | `doc/Mac版本可行性与实施.md` | 保留关键技术结论、验证记录和架构边界 |
| 开发说明 | `doc/开发者手册.md` | 说明源码运行、目录结构、平台差异 |
| 数据库说明 | `doc/数据库介绍.md` | 保留数据库背景，避免声称 Mac 4.x 已完全一致 |
| 审计/清理 | `doc/外部审计报告.md`、`doc/材料清理报告.md` | 每次材料维护后更新 |

## 4. 禁止项

- 禁止出现“100% 复刻”“生产可用”“完全等价 PyWxDump”等未被代码和测试支持的结论。
- 禁止把过程输出、测试 CSV、临时 HTML、阶段计划当作长期文档。
- 禁止推荐已知接口不匹配的命令作为主流程。
- 禁止在文档中暴露数据库 key、个人聊天正文样例或敏感联系人内容。
- 禁止要求用户修改原始 `/Applications/WeChat.app`；只能推荐复制副本重签。

## 5. 命令书写规范

- 命令必须从项目根目录运行。
- 路径统一优先使用脚本默认值：`app/Database/MacMsg`。
- 如果本机实际存在 `app/DataBase/MacMsg` 副本，文档只作为兼容说明，不作为主路径。
- 涉及全量导出的命令必须提示耗时和磁盘空间。

## 6. 状态标记

| 标记 | 含义 |
|------|------|
| 已验证 | 已在当前目录执行过，输出存在且结果符合预期 |
| 可用 | 代码存在，按接口可运行，但未在本次审计中全量执行 |
| 部分支持 | 代码存在但功能有明显边界 |
| 实验性 | 依赖环境、权限、微信版本或缓存状态，不保证稳定 |
| 当前不推荐 | 入口存在但与当前代码接口不一致或风险较高 |

## 7. 闭环要求

每次文档维护必须给出：

1. 问题清单：文档与代码不一致、互相矛盾、夸大或缺失的内容。
2. 修正动作：删除、合并、重写或新增的材料。
3. 验证记录：至少包括文档引用检查、脚本帮助检查或样例导出检查。
4. 遗留风险：代码封板不能修复的问题必须写入文档，而不是在文档中回避。
